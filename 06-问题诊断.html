智体城 - 问题诊断与修复

日期: 2026-04-26 📌 问题诊断基于 [[01-系统架构总览]] 和 [[02-AI决策框架]]

一、当前架构问题

1.1 两套独立的 WebSocket 系统

服务器 (47.77.238.56)
  ├── ws-handler.js (处理3D客户端连接, 端口9876)
  └── message-bridge.js (处理OpenClaw插件, 端口9879)
         ↓
   AIEngine - decisionLoop() 每1秒运行一次

关键问题： - message-bridge 和 ws-handler 不互通 - 两个系统的客户端收不到彼此的消息

1.2 MoveToSkill 不广播位置变化

// server/ai/skills/move-to.js
async execute(agent, params, context) {
    // 只更新数据库，不广播！
    await agentStore.updatePosition(agent.agentId, { x, z });
    return { success: true, position: { x, z } };
    // 3D 客户端永远不会收到 AGENT_MOVED 广播
}

二、目标架构

1. 智体城服务器收集环境信息
2. 通过 WebSocket/MessageBridge 发送给 OpenClaw Agent
3. OpenClaw Agent 调用自身 LLM 思考并决策
4. 决策发回给智体城
5. 智体城执行决策，广播结果给 3D 客户端

三、问题定位总结

问题 位置 后果
MoveToSkill 不广播 server/ai/skills/move-to.js AI 移动后 3D 客户端不知道
message-bridge 和 ws-handler 不互通 server/ 两个系统的消息隔离
OpenClaw Agent 没有主动获取环境信息 agent-city-channel.js 只能被动响应消息

四、修复方案

4.1 立即修复：让 MoveToSkill 广播位置变化

在 ai-engine._makeDecisionInternal() 执行完 skill 后，广播 AGENT_MOVED

if (skillName === 'move_to' && result.position) {
    broadcastToAllClients({
        type: 'AGENT_MOVED',
        agentId: agent.agentId,
        position: result.position
    });
}

4.2 中期修复：打通 message-bridge 和 ws-handler

让 message-bridge 在收到 Agent 的决策消息时，通知 ws-handler 广播给 3D 客户端。

4.3 长期：实现用户描述的架构

让 OpenClaw Agent 主动向智体城请求环境信息。
